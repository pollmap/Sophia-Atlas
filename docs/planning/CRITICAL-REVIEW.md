# 기획서 비판적 리뷰 (Critical Review)

> 10개 기획서에 대한 냉정한 분석. 과대 설계, 현실성 부족, 누락된 사항을 지적한다.

---

## 종합 평가

### 강점
- 데이터 스키마(01)가 탄탄함. Person/Entity/Relationship 3축 모델은 적절
- 시각화 설계(06)의 D3.js Canvas 전환점(500노드)과 성능 예산이 현실적
- 기술 스택(09)의 ADR(Architecture Decision Records)이 의사결정 근거를 명확히 기록

### 핵심 문제: 규모 대비 실행력 불균형

**가장 큰 위험은 "기획의 크기"와 "실행 가능한 자원"의 괴리다.**

현재 상황:
- 개발자: 사실상 1명 (AI 보조)
- 인물 데이터: 32명 (목표 573명, 최종 1,000명+)
- 기획서: 7,783줄, 10개 문서

573명의 인물 데이터를 "학술적 수준"으로 작성하는 것은 전문 연구자 팀이 수개월 걸리는 작업이다. AI로 초안을 생성할 수 있으나, 팩트 체크 없는 데이터는 프로젝트의 신뢰도를 근본적으로 훼손한다.

---

## 문서별 비판

### 01. 데이터 아키텍처 & 온톨로지

**문제점:**
1. **Person 스키마가 과도하게 복잡**: 30+ 필드를 가진 인물 데이터를 573명분 작성하는 것은 비현실적. 필수 필드를 10개 이하로 줄이고 나머지는 점진적 보강이 현실적.
2. **관계 23종은 너무 세분화**: 실제로 `influenced`와 `developed`를 정확히 구분할 수 있는가? `parallel`과 `contextual`의 차이가 사용자에게 의미 있는가? 8~12종으로 축소 권장.
3. **음수 연도 처리** (BC): `period.start: -470`은 JSON에서 정상이지만, 타임라인 시각화에서 BC/AD 전환점 처리 로직이 명시되지 않음.

**개선안:**
- 필수 필드를 `id, name, era, period, location, category, summary` 7개로 확정
- 관계 유형을 `influenced, opposed, parallel, teacher_student, collaborated, contextual, founded, authored, belongs_to` 9종으로 축소
- BC/AD 타임라인 렌더링 전략 추가

### 02. 인터랙션 & 애니메이션 설계

**문제점:**
1. **애니메이션 토큰이 과잉 설계**: 현재 단계에서 8종의 duration 토큰과 12종의 easing이 필요한가? CSS 기본값(`transition: 200ms ease`)으로 충분한 단계.
2. **prefers-reduced-motion 대응**은 좋으나, 현재 프로젝트에 애니메이션이 거의 없는 상태에서 이 수준의 설계는 시기상조.
3. **D3.js 인터랙션 코드**가 설계서에 포함되어 있으나, 실제 D3.js 컴포넌트가 아직 미구현 상태.

**개선안:**
- Phase 2에서는 CSS transition 기본값만 사용
- D3.js 인터랙션은 그래프 컴포넌트 구현 시 상세화
- 애니메이션 토큰은 Phase 3에서 체계화

### 03. IA/UX 네비게이션 설계

**문제점:**
1. **5개 유저 페르소나**가 정의되어 있으나, 실제 사용자 조사 없이 가정에 기반. 출시 후 분석 데이터로 검증해야 함.
2. **30개+ 페이지**의 와이어프레임이 ASCII로 제시되어 있으나, 실제 구현과 괴리가 클 수 있음.
3. **모바일 네비게이션**이 햄버거 메뉴로만 처리되어 있는데, 시각화 중심 앱에서 모바일 경험을 심각하게 고민해야 함.

**개선안:**
- 유저 페르소나는 "가설"로 명시, 출시 후 검증 계획 추가
- 와이어프레임은 Figma/Excalidraw로 시각화 (ASCII 한계)
- 모바일에서의 그래프/타임라인/지도 경험을 별도 섹션으로 다룸

### 04. 콘텐츠 전략 & 라이팅 가이드

**문제점:**
1. **"학술적 수준의 콘텐츠"를 표방하지만, 팩트 체크 프로세스가 없음.** 누가 "소크라테스의 산파술이 플라톤 철학의 출발점"이라는 서술을 검증하는가?
2. **MVP 200명의 summary + detailed**를 누가 작성하는가? AI 생성 콘텐츠의 hallucination 위험.
3. **다국어 전략**(한국어 기본, 영문 병기)은 좋으나, 영문 summary를 누가 검수하는가?

**개선안:**
- AI 생성 콘텐츠는 "초안"으로 명시, 검증 파이프라인 필수
- 출처/참고문헌 필드를 필수로 승격
- Phase 2에서는 한국어 summary만 집중, 영문은 Phase 3

### 05. 질문 계보 시스템

**문제점:**
1. **이 기능 자체가 Phase 2의 범위인가?** 30개 근본 질문 + 전통별 경로 + 시각화는 독립 프로젝트 수준.
2. **QuestionExplorer 인터페이스**가 너무 복잡. 사용자가 "신은 존재하는가?"를 검색했을 때, 실제로 보여줄 수 있는 데이터가 있는가?
3. **현재 데이터(32명)로는 이 기능을 의미 있게 구현할 수 없음.** 최소 MVP 200명 데이터 완성 후 착수해야 함.

**개선안:**
- Phase 3 이후로 연기
- Phase 2에서는 기존 `questions` 페이지를 단순 개선하는 정도로 제한
- 질문 데이터는 인물 데이터의 `questions` 필드를 활용하는 것부터 시작

### 06. 다중 시각화 뷰

**문제점:**
1. **5개 시각화 뷰**(Force Graph, Timeline, Map, Tree, Compare)를 모두 Phase 2에서 구현하려는 것은 비현실적. 각 뷰가 독립적으로 수주일의 개발 필요.
2. **Canvas 렌더링 코드**가 설계서에 포함되어 있으나, 실제로 Canvas로 전환이 필요한 시점(500+ 노드)은 Phase 3에나 도달.
3. **React-Leaflet 지도**는 타일 서버 의존성이 있음. OpenStreetMap 타일 사용량 정책 확인 필요.

**개선안:**
- Phase 2: Force Graph(D3.js SVG) + Timeline(기존 개선) 2개만
- Phase 3: Map(Leaflet) + Tree(D3.js) 추가
- Phase 4: Compare Matrix + Canvas 최적화
- 지도 타일: Mapbox 또는 자체 타일 서버 검토

### 07. 도메인 확장 아키텍처

**문제점:**
1. **Wave 1/2/3 구분**은 좋으나, 각 Wave의 구체적 인물 목록이 없음. "관계 밀도 상위 50명"이라고만 되어 있음.
2. **동서양 균형 전략**(서양 55%, 동아시아 20%)의 근거가 불명확. 프로젝트의 정체성이 "인류 사상의 지도"인지 "서양 철학 중심 + 동양 보충"인지 명확히 해야 함.
3. **인덱스 파일 전략**은 좋으나, Next.js SSG에서 빌드 타임에 모든 데이터를 이미 로드하므로, 클라이언트 사이드 인덱스의 실효성이 제한적.

**개선안:**
- Wave 1 인물 50명의 구체적 ID 목록 확정
- 동서양 비율의 학술적 근거 또는 프로젝트 비전 명시
- SSG + 클라이언트 하이브리드 데이터 전략 재검토

### 08. 커뮤니티 & 기여 시스템

**문제점:**
1. **현재 단계에서 가장 불필요한 기획서.** 사용자가 0명인 프로젝트의 커뮤니티 거버넌스를 설계하는 것은 과잉.
2. **편집 위원회**(10명), **Discord 채널**, **CLA 봇** 등은 Phase 4 이후의 논의 사항.
3. **기여자 점수 시스템**과 **등급 체계**는 위키피디아 수준의 커뮤니티에서나 의미 있음.

**개선안:**
- Phase 2에서는 GitHub Issues만 열어두기
- CONTRIBUTING.md만 작성하면 충분
- 커뮤니티 설계는 첫 외부 기여자가 나타난 후 재검토

### 09. 기술 아키텍처 & 성능

**문제점:**
1. **가장 현실적인 문서이나, 현재 코드베이스와의 괴리가 큼.** 설계서의 data-loader.ts, graph-utils.ts, search.ts가 실제 코드와 다름.
2. **테스트 전략**(유닛/통합/E2E)이 있으나, 현재 테스트가 0개. 테스트 프레임워크 설정부터 시작해야 함.
3. **Vercel Analytics + Speed Insights**는 좋으나, 무료 플랜 한도 확인 필요.

**개선안:**
- 설계서를 현재 코드베이스에 맞게 업데이트
- 테스트는 데이터 검증 스크립트부터 시작 (E2E는 Phase 3)
- 모니터링은 Lighthouse CI만으로 시작

### 10. 아이콘 & 일러스트레이션 시스템

**문제점:**
1. **커스텀 SVG 아이콘/일러스트**는 디자이너 없이 구현 불가. Lucide 아이콘만으로 충분.
2. **종교 심볼 12개**를 직접 제작하는 것은 문화적 민감성 문제도 있음. 검증된 오픈소스 심볼셋 사용 권장.
3. **인물 초상화**(v3+)는 저작권 문제. 공개 도메인 이미지 또는 AI 생성 이미지의 법적 지위 불확실.
4. **"프레스코 판화 스타일"**이라는 일러스트 방향은 좋으나, 이를 일관되게 제작할 역량이 있는가?

**개선안:**
- Phase 2: Lucide 아이콘만 사용, 커스텀 SVG 없음
- 종교 심볼: Wikimedia Commons의 공개 도메인 SVG 활용
- 인물 초상화: Phase 4 이후, 라이선스 확보 후 진행
- 로고만 커스텀 제작 (외부 디자이너 또는 AI 도구 활용)

## 공통 문제점

### 1. 과잉 설계 (Over-Engineering)

10개 문서의 총량이 7,783줄. 현재 코드베이스 전체보다 기획서가 더 클 수 있음. "설계에 걸리는 시간 > 구현에 걸리는 시간"이 되면 역효과.

### 2. Phase 혼동

여러 문서에서 Phase 2/3/4의 범위가 불일치:
- 07에서는 "Phase 2c: 전체 573명"
- 06에서는 "Phase 2: 5개 시각화 뷰 모두 구현"
- 05에서는 "Phase 2: 질문 계보 시스템"
→ Phase별 범위의 통일된 정의 필요

### 3. 데이터 품질 vs 수량 딜레마

"MVP 200명"이라고 하지만, 200명을 Grade B 이상으로 작성하는 것은 엄청난 작업량. AI로 생성하면 수량은 달성 가능하나 품질은 보장 불가. **이 딜레마를 정면으로 인정하고 전략을 세워야 한다.**

### 4. 실제 사용자 부재

기획서 전체가 "사용자 검증 없는 가정"에 기반. 출시 후 사용자 피드백에 따라 기획의 70%가 변경될 수 있음을 인정해야 한다.

### 5. 배포 전략 미결

Vercel과 GitHub Pages 중 어느 것이 주 배포 환경인지 아직 확정되지 않음. next.config.js에서 `output: 'export'`와 `basePath` 설정이 왔다갔다 하고 있음.

## 결론

기획서의 비전은 훌륭하지만, **"인류 사상의 시공간 지도"라는 거대한 꿈과 "1인 개발 + AI 보조"라는 현실 사이의 간극이 매우 크다.**

성공 전략은 **극단적 집중**이다:
1. 50명의 인물을 완벽하게 만들어 작동하는 프로토타입을 보여주는 것이 200명의 불완전한 데이터보다 낫다
2. Force Graph 하나를 훌륭하게 만드는 것이 5개 시각화를 다 만드는 것보다 낫다
3. 기획서 10개를 쓰는 시간에 코드 10개 파일을 작성하는 것이 낫다

**지금부터는 기획이 아닌 구현에 집중해야 한다.**
